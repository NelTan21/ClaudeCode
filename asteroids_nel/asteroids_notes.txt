1. PYTHON IMPORTS — WHAT CAN AND CAN'T BE IMPORTED
--------------------------------------------------------------------

RULE: only TOP-LEVEL names in a file are importable — classes,
module-level functions, module-level variables/constants.

NOT importable directly:
  - Methods defined INSIDE a class body (e.g. `draw`, `update`).
    They aren't top-level names in the file; they only exist as
    attributes of the class (CircleShape.draw), not as standalone
    names you can `from module import draw`.
  - Variables created INSIDE a function. e.g. `screen` in main.py
    is created inside `def main():` — it only exists while main()
    is running, and is never a module-level name. `from main import
    screen` is invalid for this reason, not just bad style.

KEY INSIGHT: importing a CLASS brings in ALL of its methods
automatically, bundled together. You never import methods
separately — that's the entire point of a class (a package deal).
  from circleshape import CircleShape   # correct
  from circleshape import draw          # invalid — draw isn't a
                                         # top-level name in the file

IMPORT vs PARAMETER PASSING — two different mechanisms:
  - IMPORT = a name becomes permanently referenceable in a file,
    starting when that file loads.
  - PARAMETER = a value is handed across a function call boundary
    at the MOMENT of the call, regardless of which file the
    function/caller live in. No import needed for this.
  Example: `screen` is created in main.py inside main(), then
  passed as an argument: `Player1.draw(screen)`. player.py's
  `draw(self, screen)` receives it as a parameter — it never
  needed `from main import screen`.

CIRCULAR IMPORTS: if file A imports from file B, and file B also
imports from file A, Python chokes trying to fully load either one
first. Fix: only import a name into a file if that file's code
actually references it somewhere. (Root cause of an early bug:
circleshape.py and player.py both had `from main import main`
despite never calling `main()` — pure dead weight that also broke
the whole program.)

RULE OF THUMB: before adding an import, ask "does this file's code
literally use this name anywhere below the import line?" If not,
delete the import.

2. CLASS INHERITANCE — METHODS VS ATTRIBUTES (the big one)
--------------------------------------------------------------------

METHODS are inherited automatically the INSTANT a class is defined:
    class Asteroid(CircleShape):
This one line alone gives Asteroid access to every method
CircleShape has (draw, update, etc.) — no code needs to run, no
constructor needs to be called. It happens at class-definition time.

ATTRIBUTES work completely differently. A brand-new instance has
NO instance attributes at all until some code actually executes
`self.x = value` against it. Nothing is pre-loaded or "inherited
in the background" the way methods are.

super().__init__(x, y, radius) is what MAKES that code run:
  - It executes the PARENT's __init__ method body, against your
    object (self), using the values you hand it.
  - This is not "loading defaults" — it's literally running the
    parent's statements right now:
        self.position = pygame.Vector2(x, y)   # uses your x, y
        self.velocity = pygame.Vector2(0, 0)    # hardcoded, ignores
                                                 # your arguments entirely
        self.radius = radius                    # uses your radius
  - Note self.velocity isn't influenced by anything you passed in —
    it's just a line of parent code that always runs the same way.

DATA FLOW DIRECTION — only goes ONE way, subclass → parent:
    Asteroid(100, 200, 20)
        -> Asteroid.__init__(self, x=100, y=200, radius=20)   [your own signature]
        -> super().__init__(x, y, radius)   [you PASS/FORWARD these down]
        -> CircleShape.__init__ runs, mutates self
  __init__ never returns anything (implicitly returns None).
  "Returned" is the WRONG word here — return means a value comes
  back out to be caught (`result = func()`). PASS / FORWARD is
  correct — you're relaying a value you already have onward, with
  nothing coming back.

OVERRIDING vs PARAMETERIZING — a real distinction:
  - OVERRIDING = redefining a method's actual LOGIC (e.g. Asteroid
    writing its own draw() that replaces CircleShape's formula).
  - Supplying different VALUES to the same unchanged parent formula
    is NOT overriding — it's just normal parameterization.
  - Real overriding danger example (an actual bug hit in this
    project): if child code runs `self.position = (x, y)` AFTER
    calling super().__init__(), that DOES overwrite what the
    parent just set — replacing a proper pygame.Vector2 with a
    plain tuple that doesn't support Vector2 math (breaks `+=`).

FINAL STATE: once all __init__ code (parent's + child's) has run,
there is ONE flat attribute list on the instance. No separate
"parent attributes" vs "child attributes" — whichever assignment
ran LAST for a given name wins. This is also why a subclass
instance genuinely IS-A parent instance (isinstance(asteroid,
CircleShape) is True) — inheritance is additive on a real base
object, not a reference to something else at a distance.

3. ASSIGNMENT (=) VS ACCUMULATION (+=) — real bug hit
--------------------------------------------------------------------

Bug found in player.py:
    def rotate(self, dt):
        self.rotation = PLAYER_TURN_SPEED * dt   # WRONG

Since PLAYER_TURN_SPEED and dt are roughly constant every frame,
this line resets self.rotation to nearly the same value every
single frame — it never builds on the previous value, so the ship
never visibly kept turning past a tiny fixed offset.

FIX: use += so each frame's turn amount ADDS onto the existing
rotation, same pattern as movement:
    self.rotation += PLAYER_TURN_SPEED * dt
    self.position += rotated_with_speed_vector   # (move() already did this right)

RULE: whenever a value needs to build up frame over frame (angle,
position, score, timers), it needs +=, not =. Plain = throws away
history and pins the value to whatever that frame calculates.

4. PYGAME SPRITE GROUPS
--------------------------------------------------------------------

A pygame.sprite.Group is an ITERABLE CONTAINER — conceptually just
like a list of objects. `for obj in some_group:` works exactly like
looping a plain list. Nothing magic about the looping part.

What IS special about Group:
  a) Membership is automatic via a `containers` class attribute:
        Player.containers = (updatable, drawable)
     CircleShape.__init__ checks `hasattr(self, "containers")` and
     if present, adds the new instance to those groups automatically.

  b) ORDER MATTERS: `Player.containers = (...)` must be set BEFORE
     any `Player(...)` instances are created. The containers check
     happens ONLY at construction time — an instance created before
     `.containers` was assigned never joins the group, silently.

  c) group.update(*args) is a built-in shortcut that automatically
     calls .update(*args) on every member — replaces manually
     calling .update() on every single object by name.

  d) group.draw(surface) is ALSO built-in, but DON'T use it here —
     it expects each sprite to have .image and .rect attributes
     (the standard pygame sprite pattern), which CircleShape/Player/
     Asteroid don't have. Instead, loop manually and call each
     object's own custom draw method:
        for obj in drawable:
            obj.draw(screen)
     This is why the assignment phrased update() and draw() usage
     differently — update() leans on the built-in fan-out, draw()
     needs a manual loop because of the custom draw(self, screen)
     signature this project uses.

5. RENDER LOOP ORDER
--------------------------------------------------------------------

Correct order, every frame:
    1. screen.fill("black")      # clear the previous frame
    2. draw everything for this frame (player, asteroids, etc.)
    3. pygame.display.flip()     # present the finished frame

Bug hit: draw() was called BEFORE fill() and flip(). Sequence was
draw -> fill -> flip, so fill() painted over the just-drawn objects
before flip() ever displayed them — result was a plain black screen
even though drawing logic was otherwise correct.

Related gotcha: after fixing the order, screen.fill("white") was
left in from testing, and the triangle is drawn in "white" too —
a white shape on a white background is invisible even with 100%
correct logic. Always sanity-check fill color vs draw color.

6. UNDEFINED NAME BUGS — a recurring pattern to watch for
--------------------------------------------------------------------

Several bugs in this project came from referencing a bare name
inside a method that was never actually defined in that method's
scope (not a parameter, not a local variable, not a self.attribute):

  - super().__init__(x, y, radius) failed once when Player's
    __init__ only accepted (self, x, y) — `radius` didn't exist
    anywhere in scope.
  - Asteroid.draw() referenced a bare `radius` instead of
    `self.radius` — draw()'s only parameters are (self, screen),
    so `radius` isn't defined anywhere in that method.

CHECKLIST when a NameError shows up: for the offending name, ask —
is it a parameter of this exact method? A local variable assigned
earlier in this exact method? Or does it need to be `self.<name>`
because it's an attribute set elsewhere (e.g. in __init__)?

8. VECTOR2 MENTAL MODEL — POINT VS ARROW (the shoot() bug)
--------------------------------------------------------------------

A Vector2(x, y) is the SAME class either way — pygame doesn't tag it
as "point" or "arrow." Which one it means is entirely determined by
which variable holds it and how your code uses it:

  - self.position  -> a POINT: "where this object currently is"
  - self.velocity  -> an ARROW: "which way, and how far per step"

KEY INSIGHT: a Point class with no .rotate()/.length() would just be
dead coordinates — asking "which way does it point?" wouldn't even
make sense. What makes Vector2 a vector is the OPERATIONS attached
to the two numbers (rotate, length, +, * scalar), not merely having
two numbers.

ANGLE vs MAGNITUDE — two independent knobs, not two versions of the
same thing (a conflation that caused real confusion mid-discussion):
  - ANGLE/DIRECTION = which way the arrow points
  - MAGNITUDE       = how LONG the arrow is (speed/strength)
  .rotate(deg)   changes direction only, length is preserved.
  v * scalar     changes length only, direction is preserved.
  Example: Vector2(0,1).rotate(90) -> approx (-1, 0)  [still length 1,
  just pointing a different way]. Then * 500 -> (-500, 0) [now length
  500, same direction as the rotated vector — NOT the original (0,1)].

POSITION + VELOCITY = new POSITION:
  position is a point, velocity is an arrow. Adding them means "place
  the arrow's tail at the point; the new point is wherever the
  arrowhead lands." This is why Shot.update() moving every frame is
  simply:
        self.position += self.velocity * dt
  Re-walking a little further along the same arrow, every frame.

WHY WASD IS SPLIT THE WAY IT IS:
  self.rotation is stored as a plain NUMBER (degrees), not a vector.
  rotate(self, dt): self.rotation += PLAYER_TURN_SPEED * dt  — pure
  angle bookkeeping, no vector math at all. But move() and shoot()
  both need an actual ARROW to do position/velocity math, so both
  independently do the SAME conversion:
        Vector2(0, 1).rotate(self.rotation)
  turning the stored angle NUMBER into a usable direction VECTOR.
  Not a coincidence — it's "angle -> arrow" happening in two places
  because two different actions (moving, shooting) each need it.

INERT UNTIL UTILIZED — the core unlock of this whole discussion:
  A locally computed vector (rotated_vector, rotated_with_shoot_
  speed_vector, etc.) does NOTHING to the game by itself. It just
  sits in memory as a value. Nothing changes until that value is
  ASSIGNED into self.position or self.velocity — that assignment is
  the moment "storage" becomes "effect."

THE ACTUAL shoot() BUG, found by applying the above:
  Original code:
        new_shot.velocity = pygame.Vector2(0,1)
        rotated_vector = new_shot.velocity.rotate(self.rotation)
        rotated_with_shoot_speed_vector = rotated_vector * PLAYER_SHOOT_SPEED
        new_shot.position += rotated_with_shoot_speed_vector
  new_shot.velocity was set ONCE to the un-rotated (0,1) and never
  touched again. The carefully rotated + scaled vector was instead
  added directly to position — a single one-time nudge at creation.
  Since Shot.update() re-reads self.velocity every frame
  (position += velocity * dt), the shot would have kept drifting
  straight down every frame regardless of which way the player was
  actually facing when it fired.

  FIX (current state of player.py shoot()):
        new_shot.starting_v = pygame.Vector2(0,1)
        new_shot.velocity = new_shot.starting_v.rotate(self.rotation) * PLAYER_SHOOT_SPEED
        # new_shot.position += new_shot.velocity   (no longer needed —
        #   Shot.update() applies velocity every frame on its own)
  Now the ROTATED + SCALED vector itself becomes new_shot.velocity —
  direction AND speed both land in the one attribute Shot.update()
  actually reads every frame, instead of a one-time position nudge.

9. CLASS vs INSTANCE — main.py collision check (single-player band-aid)
--------------------------------------------------------------------

Found while reviewing this line in main.py:
    for asteroid in asteroids:
        if asteroid.collides_with(Player1) == True:

`Player1` is a hardcoded single instance, so this only ever checks
collision against ONE specific player — a band-aid that silently
breaks for any multi-player scenario.

FIRST INSTINCT (corrected during discussion): swap `Player1` for
the bare class name `Player`. This does NOT work and would crash.

KEY INSIGHT (ties directly to section 2 above): calling/naming a
CLASS does not run __init__ and does not populate instance
attributes. `Player` is the blueprint; it has never had __init__
executed against it, so it has no self.position, no self.radius.
`Player1 = Player(x, y)` is what actually runs __init__ and
produces an object WITH those attributes. `collides_with()` reads
`other.position` — calling it with the bare class `Player` instead
of an instance would throw an AttributeError, since the class
itself never carries instance data, only instances do after
construction.

CORRECT FIX (same auto-membership pattern already used for
Asteroid/Shot — see section 4): give Player its own sprite group so
EVERY instance auto-joins it, then loop over that group instead of
one hardcoded name.
    players = pygame.sprite.Group()
    Player.containers = (updatable, drawable, players)
    ...
    for asteroid in asteroids:
        for player in players:
            if asteroid.collides_with(player):
                log_event("player_hit")
                print("Game Over!")
                sys.exit()
Nested loop checks every asteroid against every player — same shape
already used in main.py for the asteroid-vs-shot check sitting right
next to it, just not yet applied to the player case as of this note.

STATUS AS OF THIS NOTE: plan confirmed correct, NOT YET applied to
main.py — still running the single-instance `Player1` version.

MINOR STYLE NOTE (same file): `if x.collides_with(y) == True:` is
redundant — collides_with() already returns a bool, so
`if x.collides_with(y):` is equivalent and idiomatic.

10. MODULE ACCESS vs OBJECT ATTRIBUTES, and split() ENCAPSULATION
--------------------------------------------------------------------

BUG FOUND: `angle1 = asteroid.random.uniform(20,50)` in main.py.
`random` is NOT an attribute of `asteroid` — nothing in Asteroid.
__init__/CircleShape.__init__ ever runs `self.random = ...`, so
`asteroid.random` doesn't exist (same undefined-name category as
section 6). Would raise AttributeError on execution.

WHY THE MIX-UP HAPPENED: the very next line,
`asteroid.velocity.rotate(angle1)`, IS a legitimate dot-chain,
because `velocity` really is an instance attribute set in
CircleShape.__init__. It's easy to pattern-match that shape onto
the wrong name.

KEY DISTINCTION: `import random` at the top of a file makes `random`
a directly usable name in that file's scope — a MODULE reference,
not something living on any particular object. You call it as
`random.uniform(...)` straight from module scope. Object dot-chains
(`asteroid.velocity...`) only work for names that were actually
assigned onto that specific instance via `self.name = ...`. Same
"was this name ever actually defined here?" checklist as section 6,
just applied to "module-level name" vs "instance attribute" instead
of two instance attributes.
  FIX: `angle1 = random.uniform(20,50)` — call the module directly,
  no `asteroid.` prefix.

split() ENCAPSULATION GAP: `asteroid.split()` was being called in
main.py's collision loop, but `split` was never defined anywhere
(not on Asteroid, not on CircleShape) — another undefined-name bug,
this time an undefined METHOD instead of a variable/module. Notably,
the full splitting logic (rotate two velocity vectors off the
original, spawn two smaller Asteroids, assign their velocities) had
already been hand-written directly inside main.py's loop, duplicating
work that `split()` was supposed to own.

FIX APPLIED — moved the logic INTO Asteroid as a real method,
matching how draw()/update()/collides_with() are already structured
(the object owns its own behavior; main.py just calls it):
    # asteroid.py
    def split(self):
        angle1 = random.uniform(20,50)
        velocity1 = self.velocity.rotate(angle1)
        velocity2 = self.velocity.rotate(-angle1)
        new_radius = self.radius - ASTEROID_MIN_RADIUS
        new_asteroid1 = Asteroid(self.position.x, self.position.y, new_radius)
        new_asteroid2 = Asteroid(self.position.x, self.position.y, new_radius)
        new_asteroid1.velocity = velocity1 * 1.2
        new_asteroid2.velocity = velocity2 * 1.2

main.py's collision block simplified down to just:
    if asteroid.collides_with(shot) == True:
        log_event("asteroid_shot")
        if asteroid.radius > ASTEROID_MIN_RADIUS:
            log_event("asteroid_split")
            asteroid.split()
        asteroid.kill()
        shot.kill()
RULE OF THUMB: if a chunk of object-specific logic is being
hand-computed inline in main.py's loop instead of called as a method,
that's usually a sign it belongs INSIDE the class as its own method —
same instinct as "why does draw() live on Asteroid instead of main.py
drawing circles by hand."
